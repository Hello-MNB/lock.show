How to release a version

Shipping a version is not one button — it's a small, repeatable recipe. Learn the seven ingredients and the orderly flow once, and every release becomes calm, safe, and reversible.

A version is a named, tested snapshot of the product that is live in production — with a record of exactly what changed and proof that it works. Not "some code got pushed." A named thing you can point to, trust, and roll back to.

e.g. rel-2026.07.09 → "signup fix, navigation contract, official channels, design-system Step 1"
What it's made of — the seven ingredients
1 · Scope

What's in it

The decision of which fixes/features this version contains. Bounded and named — not "everything." You approve the scope.

owner decides · kept small
2 · Build

The changes

The code, built by architecture — single sources of truth, not ad-hoc patches. Where the actual work lives.

on the work branch
3 · Gate

The proof it works

Automated checks that must pass before anything ships: navigation test (34 journeys) + language purity + app build + demo build.

npm run verify · red = does not ship
4 · Name

The version tag

A label so "what's live" has a name you can return to or roll back to. Pre-launch we use dates.

rel-YYYY.MM.DD
5 · Deploy

Going live

Promoting to production — the moment lock.show / app.lock.show actually serve the new code.

main → production (auto)
6 · Verify

Confirm it's live

Checking the real site shows the change — "live," not just "pushed." The step people skip and regret.

fingerprints on the URL
7 · Record

Write it down

Add a row to the deploy log, update the state notes and the task board. This is the ingredient that means nothing ever gets lost between releases — the difference between order and chaos.

docs/DEPLOY-LOG.md · SESSION-MEMORY.md · TASK-STATUS-BOARD.md

If something breaks

Rollback in minutes

Every version is a named tag, and Vercel keeps every past deployment. A bad release is never a crisis: re-point production to the previous deployment, or reset main to the last good tag and redeploy. This is why we can ship small and often — nothing is ever a one-way door.

The one thing still to switch on: production auto-deploy from main (the Vercel "Production Branch = main" setting, on both projects). Until that's set, step 4 doesn't reliably reach production — it's the last piece that makes this whole flow trustworthy end-to-end.

What makes a good version
→Small & frequent beats big & rare. Fewer things per version = easier to verify and roll back.
→One theme per version. "The signup + nav release," not a grab-bag.
→Always green. The gate is non-negotiable. A red gate never ships, no exceptions.
→Always recorded. If it isn't in the deploy log, it didn't happen. That's how threads stay found.
