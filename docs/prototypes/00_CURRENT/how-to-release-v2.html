How to Release a Version

The rule this document exists for: every version is documented — its status, its content, and what the NEXT version holds — per SITE, per APP, and per ENTITY (system user) — so that when something breaks we know exactly what changed, for whom, on which screen, and how to roll it back.

2 · The release pipeline (the train — every version rides it)

1
Canon-lock. Open vocabulary/product rulings answered by Maria BEFORE building. Nothing builds on shifting words.
2
One branch = one train. All site+app+docs changes ride a single branch. No partial pushes to main, ever.
3
Machine gate. npm run verify — 34 navigation journeys · language purity · app build · demo build — plus site next build. Red = train stops.
4
Preview proof. One preview deploy (Cowork's one-time hook). The FULL QA/QC protocol (§3) runs on the preview URL. Production is never the test surface.
5
DB safety. Additive migrations (new tables/RPCs) may ride the train. Structural changes (renames/drops) NEVER ride — they are separate owner-approved migrations, only after Supabase backups are ON. Migration 021 stays FROZEN forever.
6
Atomic go-live. Merge → main = one production deploy per project. Embed (build:embed) rebuilt in the SAME push — the two surfaces must never skew.
7
Live re-verification. QA/QC protocol §3 runs again against production. Fingerprints recorded in DEPLOY-LOG.
8
Document + tag. VERSIONS.md updated · release doc written with the per-entity table · Version Map refreshed · tag pushed (Cowork/owner — integration token cannot).
9
Owner pass. Maria approves the live experience. Only then the ledger row turns fully green. A version that has not passed §3 + owner pass is NOT a valid version.

4 · The per-version documentation standard (the template — no version ships without it)

Every release doc (docs/releases/rel-*.md) contains:
① Version IDs per track (rel-app-* / rel-site-* / embed / DB head / DS version implemented) + the exact SHAs.
② The per-entity table — for EACH system user (Artist · Manager office · Production · Booker/buyer · Source Confirmer · Operator): what changed · exact screens/routes · technology behind it · account/subscription dependency · evidence of verification.
③ What the NEXT version holds for each entity (link to the locked SCOPE).
④ Status ladder per item: built → preview-verified → live-verified (evidence named) → owner-approved. A ✅ never appears without its evidence beside it.
⑤ The rollback anchor: previous SHAs per project, one line each.

5 · When something breaks — the repair playbook

1. Identify the layer. Site, app, embed, or DB? Check the Version Map §2 + DEPLOY-LOG for what shipped last and for whom (the per-entity table tells you which screens/entities are hit).
2. Roll back code in one move. Redeploy the PREVIOUS SHA (recorded per project in DEPLOY-LOG) via the deploy hook — site and app roll back independently; rebuild the embed if the app rolled back.
3. Never "roll back" the DB casually. Additive migrations don't need rollback. Structural issues = restore from backups (Supabase Pro) + owner decision. Migration 021 is FROZEN — applying it breaks live claim inserts.
4. Call the owner of the lane (v1.6.1 ownership map): design fault → Codex · code fault → Claude · flow/API/infra fault → Cowork · approval/product call → Maria.
5. Document the incident in DEPLOY-LOG (what broke, for which entity, rollback SHA, fix version) — the next version's release doc references it.
