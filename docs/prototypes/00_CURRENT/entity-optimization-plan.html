Canonical ruling · PENDING MARIA'S APPROVAL

Three customer workspace families — everything else is a role or a context, not a workspace

GPT architecture audit and Codex role-language audit both independently reach the same model. A workspace is an operating context (data · billing · seats). Only three customer families own one. Everyone else is either a scoped role assignment inside a workspace, or a non-workspace context.

Artist · אקט
Private growth/Radar context. Holds Acts — each Act its own evidence + פספורט.
Representation · ייצוג
ONE workspace, role-adapted. מנהל אמן (manager) vs איש בוקינג (booking agent) = roles, not separate screen-sets.
Production · משרד הפקה
Event-first workspace. Events + lineups; team is a setting.

Non-workspace contexts: Buyer / מזמין הופעות (reads a Passport link — can be a private individual; needs NO login) · Source-Confirmer / מאשר-מקור (accountless one-claim magic link). Operator = internal console. All other ecosystem roles (publicist, label, lawyer, roster/finance staff…) = scoped role assignments / access grants, never new workspaces. Role + authority (Act / territory / category / action / effective-dates) is assigned inside a workspace — never encoded as a permanent profiles.role identity.

The layered model

Entity family → organizational form → role assignment → authority → Act / territory / period.
Customer workspace families (own a workspace = data · billing · seats):
• Artist · אקט — private growth/Radar context; holds Acts, each with its own evidence + פספורט (per-Act, non-transferable, passport_version.act_id).
• Representation · ייצוג — ONE workspace, role-adapted: מנהל אמן (manager) vs איש בוקינג (booking agent) are roles inside it, not separate screen-sets. Solo OR team. Membership ≠ ArtistAccess.
• Production · משרד הפקה — event-first workspace; Events + lineups; team is a setting.

Non-workspace contexts (by default no workspace, no shell):
• Buyer · מזמין הופעות — reads a Passport link; can be a PRIVATE individual (a wedding couple IS a מזמין הופעות); public Passport review needs NO login. NOT אמרגן.
• Source-Confirmer · מאשר-מקור — accountless one-claim magic-link task; NO signup, NO dashboard, never built as a workspace; only /confirm/:token.

Internal: Operator — the /admin console, granted by runbook.

Everything else (publicist, label, lawyer, roster/finance/territory staff…) = scoped role assignments / access grants inside a workspace, never new top-level workspaces. Role + authority (Act / territory / category / action / effective-dates) is assigned inside a workspace — never encoded as a permanent profiles.role identity.

Codex — every screen must answer six questions

  • Who are you? — the Person (login), never the public identity.
  • What form? — individual / team / company.
  • What role now? — the effective role in the current workspace (derived, not a permanent identity).
  • For which Act / opportunity / territory? — the scope of the current context.
  • What authority? — Act / territory / category / action / effective-dates.
  • What next decision? — the one primary action this screen exists to enable.

Every optimized screen above is designed to answer these — e.g. Production's event-first reorg answers "what next decision?" with an event, not a member list; the Representation roles panel answers "what role now / what authority?"; the Buyer context answers "who are you / what form?" without forcing a workspace.
