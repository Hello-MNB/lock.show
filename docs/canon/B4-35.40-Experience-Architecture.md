3A. LOCALE AND DIRECTION CONTRACT

The current product architecture contains English, Hebrew, Russian and German. English is the first implementation and source-language baseline. Hebrew uses native RTL; English, Russian and German use native LTR. Locale selection is explicit and persistent; no locale is forced solely by country.

Switching locale preserves the route, active role, organization and artist context, entered data, drafts, saved state and recoverable progress. All locales use the same persisted records and permission rules.

The Translation Matrix governs keys, semantic parity, copy state, native review and activation. Internal routes, schema keys and analytics event names remain locale-neutral implementation identifiers. Visible labels, help, receipts and transactional copy are localized for every activated locale.

The public Bookability Passport may open in the shared-link locale or saved preference. Changing locale does not change proof meaning, support, method or PassportVersion.

Russian and German routing and dictionaries are prepared during Beta but remain hidden, unpublished and noindex until native review and Maria approval. Arabic is not a current interface locale; user-entered Arabic uses safe bidirectional isolation.

18\. DETAILED ANCHOR JOURNEY — ARTIST EVIDENCE TO PASSPORT

Trigger: the artist wants to create a first credible Passport, strengthen one proof area, prepare for an opportunity or refresh stale evidence.

Goal: convert obtainable evidence into an artist-reviewed Claim, update private Radar guidance and optionally publish a safe strength.

  

Phase 1 — Goal and context

Entry: authenticated artist shell or approved artist context.

User action: select immediate professional objective and active artist.

System response: load role, organization, ArtistAccess, consent, saved progress and current evidence state.

Failure/recovery: restricted, expired or revoked context explains the next permitted recovery without leaking another artist or organization.

Exit: one active artist context and one immediate objective are explicit.

  

Phase 2 — Choose proof

User action: choose the gig, Claim purpose or Radar gap worth proving.

System response: narrow the evidence request by context and explain what the evidence can and cannot establish.

Boundary: unknown is permitted; the chosen goal changes guidance, never truth or support requirements.

Exit: one evidence purpose and related context are selected.

  

Phase 3 — Capture evidence

User action: upload an artifact or reference an approved source; provide owner authority, date, privacy, retention and withdrawal information.

System response: validate type and authority, preserve the original artifact and start processing.

Required alternate states: validation failure, interrupted upload, delayed processing, failed processing, retry, replacement and withdrawal.

Exit: an EvidenceArtifact exists in a safe state, even when processing fails.

  

Phase 4 — Process and review Claim

User action: inspect the proposed buyer-readable Claim and approve, correct, reject, hide, replace support or request reprocessing.

System response: show Claim, context, method, reviewed date, support state, freshness, private value where permitted and proposed public representation.

Trust boundary: AI never publishes; unsupported proposals remain artist-declared, not assessable or rejected.

Exit: the Claim has an explicit artist decision and traceable permitted support.

  

Phase 5 — Radar update

User action: understand what is known, missing, stale and worth doing next.

System response: refresh linear diagnostic cards, bounded evidence states, freshness, Passport readiness and one prioritized next action.

Boundary: no aggregate score, ranking or score-shaped visualization.

Exit: the artist can explain the current state and next useful action without founder interpretation.

  

Phase 6 — Select and preview

User action: choose passport-ok Claims, record separate publication consent and inspect exact public output.

System response: use the same PassportRenderer as the public page and expose exact blockers with corrective routes.

Failure/recovery: declined publication consent keeps lawful private product use available; blocked preview preserves selection and links to the required task.

Exit: an approved draft PassportVersion is ready or a bounded blocker is explicit.

  

Phase 7 — Publish and share

User action: publish, create a share link, unpublish or later refresh.

System response: create an immutable PassportVersion, publication event and privacy-safe share state.

Boundary: public output excludes raw evidence, private values, internal confidence, gaps, recommendations and private contacts.

Exit: an active public version exists or the public link is safely unavailable after unpublish.

  

Core analytics: context\_selected; claim\_purpose\_selected; evidence\_started; validation\_failed; processing\_started; claim\_approved/corrected/rejected; radar\_opened; next\_action\_started/completed; passport\_candidate\_selected; preview\_opened; passport\_published/shared/unpublished.

21\. PRODUCT CONTRACT CLOSURE — WAVE 1

Invitation purposes are distinct: account/beta invitation, OrganizationMembership invitation and ArtistAccess invitation. Acceptance creates only the record named by the invitation. Decline, expiry, revocation, suspension and dispute preserve the correct prior context and never reveal unauthorized record existence.

  

Professional action semantics are locked:

• Save creates a private durable saved-Passport state and a distinct activity event; it is not endorsement or commercial intent.

• Forward, future fit, request specific proof and not fit create ProfessionalReaction only.

• Check availability and request price/details create ProfessionalReaction plus AvailabilityRequest.

• No action creates a booking, hold, negotiation, contract or settlement record.
