1\. LOCALIZATION PRINCIPLES

• Localize meaning, workflow and scene language; d​o not translate word for word.

• English may be authored first as the development and StringKey baseline. Hebrew is authored native-first, not translated from English. Russian and German are Phase-2 scaffolds until native-editor review; no locale is a blind word-for-word or machine-translation layer.

• Use neutral grammatical constructions by default. Gendered copy is optional and only used when the user voluntarily provides the relevant preference.

• Anonymous public viewers always receive neutral copy.

• Internal routes, event names, schema keys and semantic identifiers remain English. Visible UI labels, help text and transactional copy are localized per active locale.

• Every route activated in a locale m​ust have complete Translation Matrix coverage and functional parity for that locale. Locale switching m​ust preserve route, form state and saved progress. User-entered mixed-script content uses safe bidirectional isolation.

• Russian and German are Phase-2 UI locales. Their technical scaffolds may exist earlier, but activation requires complete matrix coverage, native-editor review, functional parity and QA; machine-translated or incomplete copy is n​ever activated automatically.

• Internal codes, maturity labels, experiment gates and methodology terminology n​ever appear in user-facing copy.

1A. TRANSLATION MATRIX — R​EQUIRED CONTROL

The Translation Matrix is a mandatory row-level control, not a prose concept. Every user-visible label, instruction, validation message, empty state, error, receipt, notification, consent string and transactional message m​ust have one stable StringKey and one matrix row before locale activation.

R​equired row fields:

StringKey; Screen ID and route; component and state; user job/context; English development-baseline string; Hebrew native-authored string; Russian string; German string; variables/placeholders; tone and p​rohibited interpretation; direction; length constraint; legal/privacy dependency; status per locale; native reviewer; accountable owner; source version; reviewed date; release version; activation flag; notes.

Permitted locale statuses:

SOURCE READY; DRAFT; NATIVE REVIEW; LEGAL REVIEW; QA PASS; ACTIVE; BLOCKED; DEPRECATED.

Control rules:

• English may be written first as the development baseline and StringKey reference, but English completion does not redefine product scope and does not make Hebrew a translation derivative.

• Hebrew is authored native-first. Russian and German require native-editor review before ship; machine translation n​ever activates a locale automatically.

• Missing translations may fall back to English only in internal development or an explicitly approved restricted test. A production locale c​annot silently mix untranslated UI.

• Variables, IDs, prices, dates, names and links are tested for placeholder integrity and bidirectional safety.

• The matrix m​ust preserve semantic parity, not literal wording. Each locale may use native scene language while keeping the same Product rule and outcome.

• The matrix may be stored in the repository or a controlled native Google Sheet, but B4-35.20 m​ust always record its canonical location, owner and version. It m​ay not be deleted or “deduplicated” into prose.

Current matrix artifact location: LOCATION UNVERIFIED. R13 and R09 m​ust identify the existing source/code matrix, register its canonical URL or repository path here and reconcile it against the four-locale decision before a non-English release verdict. This status does not authorize creating a parallel uncontrolled matrix.
